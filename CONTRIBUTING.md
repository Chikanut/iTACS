# Contributing

## Принципи

- Підтримуємо мінімальні, цілеспрямовані зміни.
- Не змішуємо feature-роботу з масовим cleanup без окремої причини.
- Якщо змінюється auth/session flow або developer workflow, обов'язково оновлюємо документацію.

## Базовий процес

1. Оновіть гілку та перевірте `git status`.
2. Встановіть залежності через `flutter pub get`.
3. Реалізуйте зміни з урахуванням існуючих сервісів і глобальних singleton-ів.
4. Відформатуйте змінені Dart-файли.
5. Прогоніть:

```bash
flutter analyze
flutter test
```

6. Переконайтесь, що в diff немає build-артефактів, backup-папок і локальних звітів.

## Стиль змін

- Нові docs-файли пишемо українською.
- Для auth/session логіки використовуємо `AppSessionController` і `AuthService`.
- Для Firestore-операцій не додаємо прямі виклики в UI, якщо вже є відповідний сервіс.
- Для великих refactor-ів спочатку фіксуємо технічний борг у `docs/ROADMAP.md`.

## Що перевіряти перед merge

- persisted login не змушує користувача повторно входити без явного logout;
- logout очищає локальний профіль і повертає застосунок до `LoginPage`;
- нові файли або команди описані в `README.md`, якщо це впливає на onboarding.
