```bash
firebase emulators:exec --only firestore "node test/rules.test.js"
```

Expected output: All tests pass (exit 0). Each test should show "pass" from Node.js built-in test runner output.

**If any test fails:**

1. Read the failure output carefully. Each failure will name the test (e.g., "viewer reads allowed_users — ALLOW") and the error type.

2. For `assertFails` failures (expected DENY but got ALLOW): the rule is too permissive. The fix is in `cloudstore_rules` — tighten the allow condition for that collection/operation.

3. For `assertSucceeds` failures (expected ALLOW but got permission-denied): the rule is too restrictive. Either:
   - The seeded data doesn't match what the rule expects (fix the seed in helpers.js or the before() hook in rules.test.js)
   - The rule condition has a bug (fix cloudstore_rules)

4. For test infrastructure failures (module not found, emulator connection refused):
   - Module not found: ensure `process.chdir(__dirname)` is at the top of rules.test.js, and `test/node_modules` exists
   - Connection refused: the emulator may need a moment; retry with `--export-on-exit=/tmp/emulator-data` or check port 8080 is free

5. After each fix to `cloudstore_rules`, re-run `firebase deploy --only firestore:rules` before re-running tests (the emulator reads the rules file fresh on exec, so re-deploy to production is not strictly needed for emulator tests — the emulator reads from the file directly).

**Known expected state from 02-02:**
- If outcome was A or C: files read (in-group) should ALLOW, files read (out-of-group) should DENY, files read (no groupId) should DENY, files write by editor should ALLOW, files write by viewer should DENY.
- If outcome was B: files read by any auth user ALLOW, files read by unauthenticated DENY, files write by editor ALLOW, files write by viewer DENY.

Do NOT modify any test expectations to make tests pass. Fix either the rules or the seed data.
